The SatLabs Group DVB-RCS MIB

卫星实验室组DVB-RCS MIB

Abstract

摘要

This document describes the MIB module for the Digital Video Broadcasting Return Channel via Satellite system (DVB-RCS), as defined by the SatLabs Group. It defines a set of MIB objects to characterize the behavior and performance of network-layer entities deploying DVB-RCS.

本文件描述了卫星系统（DVB-RCS）数字视频广播返回通道的MIB模块，由SatLabs集团定义。它定义了一组MIB对象来描述部署DVB-RCS的网络层实体的行为和性能。

1. Introduction
1. 介绍

The SatLabs Group [SATLABS] is an international non-profit EEIG (European Economic Interest Grouping) committed to large-scale adoption and deployment of the Digital Video Broadcasting Return Channel via Satellite (DVB-RCS) standard [ETSI-RCS]. SatLabs members are service providers, satellite operators, system integrators, terminal manufacturers, and technology providers with an interest in DVB-RCS.

SatLabs集团[SatLabs]是一家国际非营利EEIG（欧洲经济利益集团），致力于通过卫星（DVB-RCS）标准[ETSI-RCS]大规模采用和部署数字视频广播返回频道。卫星实验室成员是服务提供商、卫星运营商、系统集成商、终端制造商和对DVB-RCS感兴趣的技术提供商。

Since its creation in 2001, the main goal of the SatLabs Group has been to achieve interoperability between DVB-RCS terminals and systems. Therefore, the Group has defined the SatLabs Qualification Program, which provides an independent certification process for DVB-RCS terminals based on System Recommendations defined by SatLabs. To enhance product interoperability, beyond the physical-layer and MAC-layer mechanisms defined in the DVB-RCS standard, SatLabs has expanded its Recommendations in the field of DVB-RCS terminal management [SATLABS]. As part of this effort, SatLabs has specified a common Simple Network Management Protocol (SNMP) Management Information Base (MIB) for DVB-RCS terminals, which is defined in this document.

自2001年成立以来，卫星实验室组的主要目标一直是实现DVB-RCS终端和系统之间的互操作性。因此，集团定义了SatLabs鉴定计划，该计划根据SatLabs定义的系统建议为DVB-RCS终端提供独立的认证流程。为了增强产品互操作性，除了DVB-RCS标准中定义的物理层和MAC层机制外，SatLabs还扩展了其在DVB-RCS终端管理领域的建议[SatLabs]。作为这项工作的一部分，SatLabs为DVB-RCS终端指定了一个通用简单网络管理协议（SNMP）管理信息库（MIB），该信息库在本文档中定义。

A DVB-RCS terminal is denoted as a Return Channel Satellite Terminal (RCST) in the remainder of this document. This consists of an Indoor Unit (IDU) and an Outdoor Unit (ODU) connected through an Inter-Facility Link (IFL), usually a coaxial L-band interface. On the user side, the IDU is connected to the user network through a Local Area Network (LAN) interface (usually Ethernet). On the network side, the ODU is connected via a satellite link (the air interface).

DVB-RCS终端在本文件其余部分中表示为返回信道卫星终端（RCST）。这包括通过设施间链路（IFL）连接的室内机（IDU）和室外机（ODU），通常是同轴L波段接口。在用户端，IDU通过局域网（LAN）接口（通常为以太网）连接到用户网络。在网络端，ODU通过卫星链路（空中接口）连接。

The SatLabs Group DVB-RCS MIB is implemented in the IDU of an RCST. RCST management can be performed either through the LAN interface (local management) or through the air interface (remote management from the Network Control Center, NCC). RCST and NCC elements are shown on Figure 1.

2.2. Glossary
2.2. 术语汇编

The terms in this document are derived either from DVB-RCS standard specifications [ETSI-RCS] or from SatLabs System Recommendations [SATLABS].

本文件中的术语来源于DVB-RCS标准规范[ETSI-RCS]或SatLabs系统建议[SatLabs]。

2.2.1. Star DVB-RCS Network
2.2.1. 星型DVB-RCS网络

This denotes a hub-and-spoke configuration where all communications pass through a central hub, which usually also includes the NCC. Peer-to-peer communication between RCSTs is possible through a double satellite hop (this traffic has to pass through the hub).

这表示所有通信都通过中央集线器（通常还包括NCC）的中心辐射配置。RCST之间的点对点通信可以通过双卫星跳（该通信量必须通过集线器）。

2.2.2. Mesh DVB-RCS Network
2.2.2. 网状DVB-RCS网络

This denotes a mesh configuration that supports peer-to-peer communications in a single satellite hop directly between RCSTs.

这表示网状结构配置，支持RCST之间直接在单个卫星跳中进行对等通信。

2.2.3. Transparent DVB-RCS Network
2.2.3. 透明DVB-RCS网络

This denotes a network using transparent satellite transponders. Star or mesh network configurations can be supported. In the case of a mesh configuration, RCSTs need to incorporate a TDMA receiver in addition to the TDM receiver.

这表示使用透明卫星转发器的网络。可以支持星形或网状网络配置。在网状结构的情况下，RCST除了TDM接收器外，还需要包括TDMA接收器。

2.2.4. Regenerative DVB-RCS Network
2.2.4. 再生式DVB-RCS网络

This denotes a network that uses regenerative satellite transponders, i.e. includes some On-Board Processing functionality, which allows demodulation and decoding of the uplink TDMA signals and re-multiplex of the traffic into TDM signals on the downlink. Star or mesh network configurations can be supported.

这表示使用再生卫星转发器的网络，即包括一些车载处理功能，其允许对上行链路TDMA信号进行解调和解码，并在下行链路上将业务重新复用为TDM信号。可以支持星形或网状网络配置。

2.2.5. DVB-RCS MAC Layer
2.2.5. DVB-RCS MAC层

The DVB-RCS MAC layer represents the air interface of an RCST, as specified in [ETSI-RCS]. The interface is bi-directional and supports IP traffic over hub-spoke (star) and mesh satellite network topologies.

DVB-RCS MAC层表示RCST的空中接口，如[ETSI-RCS]所述。该接口是双向的，支持通过中心辐射（星形）和网状卫星网络拓扑进行IP通信。

2.2.6. DVB-RCS TDM
2.2.6. DVB-RCS时分复用

The DVB-RCS TDM corresponds to the forward link of a DVB-RCS transparent system or the downlink of a DVB-RCS regenerative system. It is based on either the DVB-S or DVB-S2 standard, specified in [ETSI-DVBS] and [ETSI-DVBS2] respectively. In the DVB-RCS context, this interface is uni- or bi-directional, as it may also be used for a return channel dedicated to a single terminal.

DVB-RCS TDM对应于DVB-RCS透明系统的前向链路或DVB-RCS再生系统的下行链路。它基于[ETSI-DVBS]和[ETSI-DVBS2]中分别规定的DVB-S或DVB-S2标准。在DVB-RCS上下文中，该接口是单向或双向的，因为它也可用于专用于单个终端的返回信道。

2.2.7. DVB-RCS TDMA
2.2.7. DVB-RCS时分多址

The DVB-RCS TDMA corresponds to the return or mesh link of an RCS transparent system or the uplink of an RCS regenerative system. It is specified in [ETSI-RCS].

DVB-RCS TDMA对应于RCS透明系统的回路或网状链路或RCS再生系统的上行链路。在[ETSI-RCS]中有规定。

In the context of star transparent and mesh regenerative DVB-RCS systems, this interface is uni-directional.

在星形透明和网状再生DVB-RCS系统中，该接口是单向的。

In the context of mesh transparent DVB-RCS systems, this interface is bi-directional.

在网状透明DVB-RCS系统中，该接口是双向的。

2.2.8. IDU
2.2.8. IDU

This is the indoor part of the RCST (including at least the power supply, and usually also the modem and networking functions).

这是RCST的室内部分（至少包括电源，通常还包括调制解调器和网络功能）。

2.2.9. ODU
2.2.9. ODU

This is the outdoor part of the RCST (including at least the aerial, and usually also the LNB and BUC).

这是RCST的室外部分（至少包括天线，通常也包括LNB和BUC）。

2.2.10. RCST
2.2.10. RCST

This is the Satellite Terminal, installed on the customer premises. It is composed of the IDU and ODU.

这是安装在客户场所的卫星终端。它由IDU和ODU组成。

2.2.11. NCC
2.2.11. NCC

The NCC provides Control and Monitoring Functions. It generates control and timing signals for the operation of the DVB-RCS Network.

NCC提供控制和监控功能。它为DVB-RCS网络的运行生成控制和定时信号。

2.2.12. Configuration File
2.2.12. 配置文件

The configuration file is an XML-formatted file, storing configuration parameters for the RCST and their values.

配置文件是XML格式的文件，用于存储RCST的配置参数及其值。

2.2.13. Log File
2.2.13. 日志文件

The log file is stored at the RCST. This is used to log particular events that occur on the RCST side.

日志文件存储在RCST中。这用于记录RCST端发生的特定事件。

2.2.14. Installation Log File
2.2.14. 安装日志文件

The installation log file is stored at the RCST. This logs particular events that occur on the RCST side and are related to RCST installation phase.

安装日志文件存储在RCST中。记录RCST侧发生的与RCST安装阶段相关的特定事件。

2.2.15. Antenna Alignment
2.2.15. 天线对准

This is the process to align the RCST antenna, part of the ODU, in order to enable bi-directional communication (uplink, downlink) with the satellite network.

这是一个校准RCST天线（ODU的一部分）的过程，以实现与卫星网络的双向通信（上行链路、下行链路）。

2.2.16. CW Frequency
2.2.16. 连续波频率

The CW frequency is the frequency of a Continuous Wave signal. It is a narrowband carrier transmitted for the duration of measurements during the installation of an RCST.

CW频率是连续波信号的频率。它是在安装RCST期间在测量期间传输的窄带载波。

2.2.17. Request Class
2.2.17. 请求类

A Request Class (RC) is a representation of a Per-Hop Behavior (PHB) at the MAC layer. It defines a behavior of the MAC layer for a given aggregation of traffic. This behavior includes a combination of Capacity Categories associated to the RC and a priority with respect to the other RCs supported by an RCST.

请求类（RC）是MAC层每跳行为（PHB）的表示。它为给定的流量聚合定义MAC层的行为。该行为包括与RC相关联的容量类别和相对于RCST支持的其他RCs的优先级的组合。

2.2.18. Channel ID
2.2.18. 通道ID

Each Request Class is identified by a unique Channel_ID in the communication between the RCST and the NCC.

在RCST和NCC之间的通信中，每个请求类别由唯一的通道ID标识。

2.2.19. ATM Profile
2.2.19. ATM配置文件

The ATM profile is one of the two profiles for traffic burst format on a DVB-RCS TDMA. It is based on one or more concatenated ATM cells, each of length 53 bytes, plus an optional prefix.

ATM配置文件是DVB-RCS TDMA上流量突发格式的两个配置文件之一。它基于一个或多个连接的ATM信元，每个信元的长度为53字节，外加一个可选前缀。

2.2.20. MPEG Profile
2.2.20. MPEG配置文件

The MPEG profile is one of the two profiles for traffic burst format on the DVB-RCS TDMA. It is based on a number of concatenated MPEG2-TS packets, each of length 188 bytes.

MPEG配置文件是DVB-RCS TDMA上流量突发格式的两个配置文件之一。它基于大量串联的MPEG2-TS数据包，每个数据包的长度为188字节。

2.2.21. PID Pool
2.2.21. PID池

For the MPEG profile, several RCs may be mapped within a pool of several PIDs to allow cross-RC Section Packing [ETSI-DAT]. Section Packing can be used on all PIDs and higher priority traffic can always preempt lower priority streams. This reduces the need for padding.

对于MPEG简档，可以在多个pid的池中映射多个RCs，以允许跨RC部分打包[ETSI-DAT]。节打包可用于所有PID，高优先级流量始终可以抢占低优先级流。这减少了填充的需要。

2.2.22. Capacity Categories
2.2.22. 容量类别

The TDMA timeslot allocation process for the DVB-RCS uplink supports several Capacity Categories.

DVB-RCS上行链路的TDMA时隙分配过程支持多种容量类别。

The Capacity Categories CRA, RBDC, and A/VBDC, when authorized for an RC, have to be configured from the NCC. Their configuration parameters are used to inform the RCST of the configuration of each category at the NCC side and thus help in Capacity Requests computation.

当为RC授权时，必须从NCC配置容量类别CRA、RBDC和A/VBDC。其配置参数用于通知RCST NCC侧各类别的配置，从而帮助计算容量请求。

The categories are treated independently for each RC. A SatLabs optional feature is defined that allows their configuration at the RCST level in addition to configuration per RC. This feature is denoted RCST_PARA.

对于每个RC，类别被单独处理。定义了一个SatLabs可选功能，除了每个RC的配置外，还允许在RCST级别进行配置。该特征表示为RCST_段。

2.2.22.1. Continuous Rate Assignment (CRA)
2.2.22.1. 连续费率分配（CRA）

CRA is a rate capacity that is provided in full in a continuous manner to the RCST while required.

CRA是一种在需要时以连续方式向RCST全额提供的费率容量。

2.2.22.2. Rate-Based Dynamic Capacity (RBDC)
2.2.22.2. 基于速率的动态容量（RBDC）

RBDC is a rate capacity that is requested dynamically by an RCST. RBDC is provided in response to explicit requests from the RCST to the NCC, such requests being absolute (i.e., corresponding to the full rate currently being requested). Each request overrides all previous RBDC requests from the same RCST and is subject to a maximum rate limit.

RBDC是RCST动态请求的速率容量。RBDC是响应RCST向NCC发出的明确请求而提供的，此类请求为绝对请求（即，对应于当前请求的完整费率）。每个请求都会覆盖来自同一RCST的所有先前RBDC请求，并受到最大速率限制的约束。

2.2.22.3. Volume-Based Dynamic Capacity (VBDC)
2.2.22.3. 基于卷的动态容量（VBDC）

VBDC is a volume capacity that is requested dynamically by an RCST. VBDC is provided in response to explicit requests from the RCST to the NCC, such requests being cumulative (i.e., each request adds to all previous requests from the same RCST).

VBDC是由RCST动态请求的卷容量。VBDC是为响应RCST向NCC发出的明确请求而提供的，此类请求是累积的（即，每个请求加上来自同一RCST的所有先前请求）。

2.2.22.4. Absolute Volume-Based Dynamic Capacity (AVBDC)
2.2.22.4. 基于绝对卷的动态容量（AVBDC）

AVBDC is a volume capacity that is requested dynamically by an RCST. This capacity is provided in response to explicit requests from the RCST to the NCC, such requests being absolute (i.e., this request replaces the previous ones from the same RCST).

AVBDC是由RCST动态请求的卷容量。该容量是为了响应RCST向NCC发出的明确请求而提供的，此类请求是绝对请求（即，该请求取代了同一RCST先前发出的请求）。

The combination of AVBDC and VBDC is seen as a single Capacity Category, denoted A/VBDC.

AVBDC和VBDC的组合被视为单个容量类别，表示为a/VBDC。

2.2.22.5. Population ID
2.2.22.5. 人口ID

This defines a group of RCSTs within a network.

这定义了网络中的一组RCST。

2.2.23. Start Transponder
2.2.23. 启动应答器

This is the satellite transponder on which the communication is initiated from an RCST point of view when in the installation mode. The parameters corresponding to this transponder (satellite orbital position, frequency, etc.) are stored at the RCST as power-up configuration data.

这是在安装模式下从RCST角度启动通信的卫星收发器。与该转发器相对应的参数（卫星轨道位置、频率等）作为通电配置数据存储在RCST中。

2.2.24. DVB-S
2.2.24. DVB-S

DVB-S is the Digital Video Broadcast over Satellite [ETSI-DVBS]. It is a framework and set of associated standards published by ETSI for the transmission of video, audio, and data, using the ISO MPEG-2 standard [ISO-MPEG], over satellite links.

DVB-S是通过卫星进行的数字视频广播[ETSI-DVBS]。它是ETSI发布的一个框架和一组相关标准，用于通过卫星链路使用ISO MPEG-2标准[ISO-MPEG]传输视频、音频和数据。

2.2.25. DVB-S2 and CCM/VCM/ACM
2.2.25. DVB-S2和CCM/VCM/ACM

DVB-S2 is the Second Generation of the Digital Video Broadcast for Satellite applications standard [ETSI-DVBS2]. It is a framework and set of associated standards published by ETSI for the transmission of video, audio, and data.

DVB-S2是第二代卫星应用数字视频广播标准[ETSI-DVBS2]。它是ETSI为视频、音频和数据传输发布的一个框架和一套相关标准。

BBFRAME: The main framing unit of the DVB-S2 protocol stack.

BBFRAME：DVB-S2协议栈的主要成帧单元。

CCM: In CCM transmission mode, the forward link uses a constant set of transmission parameters (FEC coding rate and modulation scheme) for all receivers.

CCM：在CCM传输模式下，前向链路对所有接收机使用一组恒定的传输参数（FEC编码速率和调制方案）。

VCM: In VCM transmission mode, the forward link uses transmission parameters that are variable on a BBFRAME-by-BBFRAME but fixed on a Receiver basis, according to fixed link and propagation conditions for each Receiver.

VCM：在VCM传输模式中，前向链路根据每个接收机的固定链路和传播条件，使用传输参数，这些参数在一帧一帧地变化，但在接收机的基础上固定。

ACM: In ACM transmission mode, the forward link uses transmission parameters that are dynamically adjusted on a BBFRAME-by-BBFRAME and Receiver-per-Receiver basis, according to actual link and propagation conditions. In order to implement ACM, feedback from each Receiver has to be provided by DVB-RCS return channel.

ACM：在ACM传输模式中，前向链路使用传输参数，根据实际链路和传播条件，在BBFRAME-by-BBFRAME和Receiver-per-Receiver基础上动态调整传输参数。为了实现ACM，每个接收机的反馈必须由DVB-RCS返回信道提供。

2.2.26. Interactive Network
2.2.26. 交互式网络

This is another name for a DVB-RCS-based satellite network.

这是基于DVB RCS的卫星网络的另一个名称。

3.4.1. DVBRCSRCST系统

The MIB objects in this group gather some basic information that would allow anyone to trace the history -- the life -- of the RCST, as well as to get a complete description of its constitution on the component point of view, including the SatLabs options/features support statement. 该组中的MIB对象收集一些基本信息，这些信息允许任何人跟踪RCST的历史（生命），以及从组件的角度获得其构成的完整描述，包括SatLabs选项/功能支持声明。许多参数将在安装时定义。

This group contains description parameters related to the RCST type (ODU type) and location. These parameters are believed to stay unchanged once it has been defined during installation. Modification of hardware equipment, maintenance operations, and geographical re-location may require an update of those MIB objects. Note that the dvbRcsRcstSystem.dvbRcsSystemLocation object gives the location of

此组包含与RCST类型（ODU类型）和位置相关的描述参数。一旦在安装过程中定义了这些参数，这些参数将保持不变。修改硬件设备、维护操作和地理位置重新定位可能需要更新这些MIB对象。请注意，dvbRcsRcstSystem.dvbRcsSystemLocation对象提供

the ODU antenna, which is needed for network operation, while the system.sysLocation (MIB-II SNMP OID) provides the location of the IDU unit, which cannot be used for the same purpose.

ODU天线是网络运行所需的，而system.sysLocation（MIB-II SNMP OID）提供IDU单元的位置，不能用于相同的目的。

The RCST must provide either Read-Write access to dvbRcsSystemOdu parameters or, alternatively, provide the list of supported devices through the dvbRcsRcstOduListGroup (see conformance section). This group of parameters, defined in the dvbRcsRcstSystem group, allows the selection by the RCST installer of the actual ODU type. RCST必须提供对dvbRcsSystemOdu参数的读写访问权限，或者通过DVBRCSRCSTODULISTGOUP提供受支持设备的列表（请参阅一致性部分）。dvbRcsRcstSystem组中定义的这组参数允许RCST安装程序选择实际ODU类型。在这种情况下，安装程序必须分别根据选择的BUC、LNB和天线设置dvbRcsOduTxType、dvbRcsOduRxType和DVBRCSODUANTENATENATYPE。

3.4.2. dvbRcsRcstNetwork
3.4.2. DVBRCSRCST网络

This group contains all the MIB objects related to network parameters.

此组包含与网络参数相关的所有MIB对象。

In this subgroup, two objects have been defined in order to differentiate between control and user traffic and associate them with a physical interface. Both dvbRcsRcstNetwork.dvbRcsNetworkLanIpAddress (Traffic) and dvbRcsRcstNetwork.dvbRcsNetworkOamInetAddress (OAM) provide the value of the IP address of, respectively, the user traffic and the control and management traffic.

在该子组中，定义了两个对象，以区分控制和用户流量，并将它们与物理接口相关联。dvbRcsRcstNetwork.dvbRcsNetworkLanIpAddress（流量）和dvbRcsRcstNetwork.dvbRcsNetworkOamInetAddress（OAM）分别提供用户流量和控制与管理流量的IP地址值。

此组包含由NCC配置RCST服务质量（QoS）的对象。

dvbRcsRequestClassTable定义了所有第2层DVB-RCS QoS参数。

3.4.7. dvbRcsFwdLink（dvbRcsFwdConfig和dvbRcsFwdStatus组）

This MIB group contains parameters that enable access to data about the forward link.

此MIB组包含允许访问有关前向链路的数据的参数。

Configuration information is kept in the dvbRcsFwdLink.dvbRcsFwdConfig subgroup. Status information is kept into the dvbRcsFwdLink.dvbRcsFwdStatus subgroup.

配置信息保存在dvbRcsFwdLink.dvbRcsFwdConfig子组中。状态信息保存在dvbRcsFwdLink.dvbRcsFwdStatus子组中。

dvbRcsFwdLink.dvbRcsFwdConfig.DVBRCSFWDSTARTABLE中的信息首次用于RCST尝试获取前向链路。所有这些对象值均与网络信息表（NIT）表[ETSI-SI]中的卫星传送系统描述符对齐。

dvbRcsFwdLink.dvbRcsFwdConfig.DVBRCSFWDStatus表中的对象与RCS映射表（RMT）[ETSI-RCS]中的卫星前向路径描述符和物理层（PL）头[ETSI-DVBS2]对齐，后者指定MODCOD（调制和FEC速率）和类型（帧长度短或长以及是否存在导频）。

配置信息保存在dvbRcsRtnLink.dvbRcsRtnConfig子组中。状态信息保存在dvbRcsRtnLink.dvbRcsRtnStatus子组中。

RCST一次只能处理一个返回链路。因此，不需要定义一个表来收集不同的SNMP对象，就像对转发所做的那样。

5. Security Considerations
5. 安全考虑

This MIB module relates to a system that allows end users to access a private network or public Internet access. As such, improper manipulation of the MIB objects represented by this MIB module may result in denial of service to a large number of end users.

此MIB模块涉及允许最终用户访问专用网络或公共互联网访问的系统。因此，对该MIB模块表示的MIB对象的不当操作可能会导致大量最终用户拒绝服务。

There are a number of management objects defined in this MIB module with a MAX-ACCESS clause of read-write and/or read- create. Such objects may be considered sensitive or vulnerable in some network environments. The support for SET operations in a non-secure environment without proper protection can have a negative effect on network operations. These are the tables and objects and their sensitivity/vulnerability:

此MIB模块中定义了许多管理对象，其MAX-ACCESS子句为read-write和/或read-create。在某些网络环境中，此类对象可能被视为敏感或易受攻击。在没有适当保护的非安全环境中支持SET操作可能会对网络操作产生负面影响。以下是表和对象及其敏感度/漏洞：

o The use of the dvbRcsNetworkNccMgtInetAddress object to specify management stations is considered only limited protection and does not protect against attacks that spoof the management station's IP address. The use of stronger mechanisms, such as SNMPv3 security, should be considered, where possible.

o 使用dvbRcsNetworkNccMgtInetAddress对象指定管理站仅被视为有限的保护，不能防止欺骗管理站IP地址的攻击。在可能的情况下，应考虑使用更强大的机制，如SNMPv3安全性。

o The dvbRcsSystemOdu objects, dvbRcsCtrlCwEnable, dvbRcsRtnConfigMaxEirp, dvbRcsRtnConfigDefIfLevel objects, and dvbRcsRcstInstall sub-tree can, if improperly or maliciously used, lead to unwanted emissions or emission levels on the satellite uplink, thereby resulting in potential degradation of the RCS service or other services using the frequency band being used.

o dvbRcsSystemOdu对象、dvbRcsCtrlCwEnable、dvbRcsRtnConfigMaxEirp、DVBRCSRTNConfigdeffleflevel对象和dvbRcsRcstInstall子树如果使用不当或恶意，可能会在卫星上行链路上导致不必要的发射或发射电平，从而导致RCS服务或使用所使用频带的其他服务的潜在降级。

o The RCST may have its configuration file changed by the actions of the management system using a combination of the following objects: dvbRcsNetworkInstallLogFileDownloadUrl, dvbRcsCtrlDownloadFileCommand, dvbRcsCtrlActivateConfigFileCommand, or dvbRcsCtrlRebootCommand. An improper configuration file download may result in substantial vulnerabilities and the loss of the ability of the management system to control the satellite terminal.

o RCST的配置文件可能会通过管理系统的操作使用以下对象的组合进行更改：dvbRcsNetworkInstallLogFileDownloadUrl、dvbRcsCtrlDownloadFileCommand、dvbRcsCtrlActivateConfigFileCommand或DVBRCSCTRLEBootCommand。不正确的配置文件下载可能会导致严重的漏洞，并导致管理系统失去控制卫星终端的能力。

o Setting dvbRcsNetworkLogFileUploadUrl to a wrong address may potentially impact debugging/troubleshooting efforts.

o 将dvbRcsNetworkLogFileUploadUrl设置为错误地址可能会影响调试/故障排除工作。

o Setting objects in dvbRcsPktClassTable could cause significant changes to default traffic filtering on an RCST.

o 在dvbRcsPktClassTable中设置对象可能会对RCST上的默认流量筛选造成重大更改。

Some of the readable objects in this MIB module (i.e., objects with a MAX-ACCESS other than not-accessible) may be considered sensitive or vulnerable in some network environments. It is thus important to control even GET and/or NOTIFY access to these objects and possibly to even encrypt the values of these objects when sending them over

在某些网络环境中，此MIB模块中的某些可读对象（即具有MAX-ACCESS而非not ACCESS的对象）可能被视为敏感或易受攻击。因此，控制甚至获取和/或通知对这些对象的访问，甚至在发送这些对象时加密这些对象的值，这一点很重要

the network via SNMP. These are the tables and objects and their sensitivity/vulnerability:

通过SNMP连接网络。以下是表和对象及其敏感度/漏洞：

o The dvbRcsNetworkNccMgtInetAddress object may provide sufficient information for attackers to spoof management stations that have management access to the device.

o dvbRcsNetworkNccMgtInetAddress对象可能为攻击者提供足够的信息，以欺骗具有设备管理访问权限的管理站。

o The dvbRcsRcstCurrentSoftwareVersion object may provide hints as to the software vulnerabilities of the RCST.

o dvbRcsRcstCurrentSoftwareVersion对象可能提供有关RCST软件漏洞的提示。

o The object dvbRcsNetworkOamInetAddress and the table dvbRcsPktClassTable may provide clues for attacking the RCST and other subscriber devices.

o 对象dvbRcsNetworkOamInetAddress和表dvbRcsPktClassTable可能提供攻击RCST和其他订户设备的线索。

SNMP versions prior to SNMPv3 did not include adequate security. Even if the network itself is secure (for example by using IPsec), even then, there is no control as to who on the secure network is allowed to access and GET/SET (read/change/create/delete) the objects in this MIB module.

SNMPv3之前的SNMP版本未包含足够的安全性。即使网络本身是安全的（例如通过使用IPsec），即使如此，也无法控制安全网络上的谁可以访问和获取/设置（读取/更改/创建/删除）此MIB模块中的对象。

It is RECOMMENDED that implementers consider the security features provided by the SNMPv3 framework (see [RFC3410], section 8), including full support for the SNMPv3 cryptographic mechanisms (for authentication and privacy).

建议实施者考虑SNMPv3框架提供的安全特性（参见[RCFC310]，第8节），包括对SNMPv3加密机制的完全支持（用于身份验证和隐私）。

Further, deployment of SNMP versions prior to SNMPv3 is NOT RECOMMENDED. Instead, it is RECOMMENDED to deploy SNMPv3 and to enable cryptographic security. It is then a customer/operator responsibility to ensure that the SNMP entity giving access to an instance of this MIB module, is properly configured to give access to the objects only to those principals (users) that have legitimate rights to indeed GET or SET (change/create/delete) them.

此外，不建议部署SNMPv3之前的SNMP版本。相反，建议部署SNMPv3并启用加密安全性。然后，客户/运营商有责任确保授予访问此MIB模块实例权限的SNMP实体正确配置为仅授予那些拥有合法权限来获取或设置（更改/创建/删除）对象的主体（用户）访问对象的权限。
